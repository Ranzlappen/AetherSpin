# Stake Engine Submission Checklist

A thorough, professional pre-submission checklist for shipping an AetherSpin title (e.g. NovaForged
v1.0) to the [Stake Engine](https://stakeengine.com) dashboard. Work top to bottom; everything must
be green before you upload.

> Two artifacts are uploaded: the **math library** (server-authoritative outcomes) and the
> **frontend production bundle** (presentation). `bash scripts/package-for-stake.sh <game>` assembles
> both into `dist-stake/<game>/`.

---

## 1. Math library

Generated by `python3 math/scripts/generate_books.py --game <game> --sims <N>` (standalone) or the
official-SDK `run.py` (certified). Located at `math/library/<game>/`.

- [ ] `books/books_base.jsonl` — one book per line for the base mode
- [ ] `books/books_bonus.jsonl` — one book per line for the buy-bonus mode
- [ ] `lookup_tables/lookUpTable_base.csv` — `id,weight,payout` (book units = multiplier × 100)
- [ ] `lookup_tables/lookUpTable_bonus.csv`
- [ ] `lookup_tables/lookUpTableIdToCriteria_base.csv` — `id,criteria`
- [ ] `lookup_tables/lookUpTableIdToCriteria_bonus.csv`
- [ ] `configs/config.json` — game id, version, provider, wincap, rtpTarget, bookAmountMultiplier, per-mode cost/RTP
- [ ] `index.json` — manifest mapping each mode to its files
- [ ] Every `payoutMultiplier` ≤ `wincapMultiplier` (the engine clamps; verify no book exceeds it)
- [ ] Book ids are unique and contiguous; weights are present for every book
- [ ] **Certified** library produced by the official SDK (millions of spins + Rust optimizer) for
      the final submission — the standalone library is for dev/staging

---

## 2. RTP / PAR documentation

- [ ] Measured **base** RTP within tolerance of `engine.rtpTarget` (96.5% for NovaForged)
      — `python3 math/scripts/validate_rtp.py --game <game> --sims 1000000 --tol 0.03` passes
- [ ] Measured **bonus** RTP within tolerance **measured against the 100× cost** (not the base bet)
- [ ] Buy-bonus is **not** player-EV-positive (RTP ≤ 100% at scale)
- [ ] **PAR sheet** generated and reviewed — `python3 math/scripts/generate_par_sheet.py --game <game> --sims 500000 --out docs/<game>-par-sheet.md`
      (headline metrics + win-distribution histogram)
- [ ] Hit frequency, free-spin frequency, win-cap frequency, max win documented
- [ ] Volatility classification stated and consistent with the distribution
- [ ] Win cap (`wincapMultiplier`) documented and enforced

---

## 3. Mandatory UI components

The web client must present, at minimum:

- [ ] **Balance** display in the player's currency (formatted via `formatCurrency`)
- [ ] **Bet levels** selector bound to `bet.levels` with the default at `bet.defaultLevelIndex`,
      respecting `minBet` / `maxBet` / `stepBet`
- [ ] **Win display** — per-spin and session win, with the payout multiplier
- [ ] **Paytable / rules** screen — symbols, payouts, paylines, feature explanations (free spins,
      multiplier wilds, expanding wilds, the ×1→×3 ladder, bonus buy)
- [ ] **Autoplay** with a stop control and sane limits
- [ ] **Bonus buy** entry (where enabled) clearly priced at `costMultiplier × bet`
- [ ] **Responsible-gaming notes** — links/affordances per jurisdiction (session reminders,
      reality checks, links to help resources)
- [ ] **Spin / stop** controls and a clear free-spins feature indicator (spins left, current multiplier)
- [ ] Error/disconnect states surfaced to the player (see §4)

---

## 4. RGS integration

- [ ] **authenticate** — establishes the session, fetches balance + currency before play is enabled
- [ ] **play** — sends the bet for the selected bet mode; amounts are integers = `dollars × 1,000,000`
- [ ] **end-round** — settles the round and finalizes the balance
- [ ] Book events are replayed **in order**; the client never recomputes the payout
- [ ] **Error handling** — network failures, auth expiry, insufficient balance, and malformed
      responses are caught and shown without corrupting balance/state
- [ ] Reconnect / resume mid-round behaves correctly (no double-settle, no lost win)
- [ ] Mock-RGS parity verified — the client behaves identically on mock books and live RGS books

---

## 5. Performance

- [ ] Sustains the target frame rate (60 FPS budget; `FRAME_BUDGET_MS = 1000/60`)
- [ ] Initial bundle and asset load within budget; assets streamed/atlased where possible
- [ ] No memory leaks across long autoplay sessions (Pixi textures/tickers disposed)
- [ ] Built with `base: "./"` so it runs from an arbitrary CDN iframe path (already set in `vite.config.ts`)
- [ ] Source maps produced for debugging; chunk sizes reviewed

---

## 6. Compliance & QA

- [ ] Game `version` bumped and consistent across `game-definition.json`, `config.json`, and the bundle
- [ ] Definition validates against `shared/schemas/game-definition.schema.json`
- [ ] Math unit tests pass (`python -m pytest math/tests -q`)
- [ ] Frontend `check` + `test` + `build` pass
- [ ] Lint/format clean (`pnpm format:check`)
- [ ] Currency formatting and localization verified
- [ ] Responsible-gaming and jurisdictional copy reviewed by compliance
- [ ] All required legal/age/RG disclosures present

---

## 7. Dashboard upload steps (version 1.0)

1. **Build the artifacts**
   ```bash
   python3 math/scripts/generate_books.py --game <game> --sims <N>   # or the official-SDK run.py
   pnpm --filter @aetherspin/frontend build
   bash scripts/package-for-stake.sh <game>                          # -> dist-stake/<game>/
   ```
2. **Create the game** in the Stake Engine dashboard; set `id`, display name, provider, and
   **version 1.0**.
3. **Upload the math library** — `books_<mode>.jsonl`, both lookup-table CSVs per mode, `config.json`,
   and `index.json` (the contents of `dist-stake/<game>/` math files).
4. **Upload the frontend bundle** — the production build from `dist-stake/<game>/`.
5. **Set RTP / config** — confirm `rtpTarget`, `wincapMultiplier`, bet modes and their costs match
   `config.json`; map each mode to its books + lookup table.
6. **Test in staging** — play through base spins, trigger and complete free spins (incl. retrigger
   and the ×1→×3 ladder), exercise the bonus buy, and confirm balance settles correctly on
   authenticate → play → end-round.
7. **Submit for certification** — attach the PAR sheet and RTP documentation; submit. Address any
   reviewer findings, bump the version if math changes, and re-upload.

### Quick gate (run before every submission)

```bash
python -m pytest math/tests -q
python3 math/scripts/validate_rtp.py --game <game> --sims 1000000 --tol 0.03
python3 math/scripts/generate_par_sheet.py --game <game> --sims 500000 --out docs/<game>-par-sheet.md
pnpm --filter @aetherspin/frontend run check
pnpm --filter @aetherspin/frontend test
pnpm --filter @aetherspin/frontend build
bash scripts/package-for-stake.sh <game>
```
