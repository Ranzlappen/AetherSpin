### The 5 Biggest Risks to Certification & Production
 1. **Idempotency Failures:** Your rgsClient.ts lacks a strict transaction/retry layer. If the frontend drops network after sending a spin request, and retries the exact same request, the RGS must recognize an idempotency key (like a UUID) and return the existing result. Otherwise, you charge the user's balance twice—a catastrophic compliance failure.
 2. **Visual Spoofing via Timing:** If your frontend increments the player's visual balance *before* the /wallet/end-round network handshake successfully resolves, and that handshake subsequently fails, you have shown the player money they didn't win. Visual completion must be tightly decoupled from server acknowledgment.
 3. **Rust vs. Python Drift:** You noted the Stake SDK uses a Rust optimizer. If your standalone Python simulator (math/simulator/) uses an RNG or line-evaluation loop that differs even slightly from how the Stake Rust engine evaluates the same reel strips, your locally generated PAR sheet is legally invalid for the production deployment.
 4. **Missing End-to-End Visual Verification:** Because the frontend generates visuals procedurally off events, changes to Svelte components or PixiJS stages can break the alignment of win-lines without failing Vitest. You need a headless visual regression tool (like Playwright with Percy or direct snapshotting) testing the bookPlayer.ts outputs.
 5. **CSPRNG vs. PRNG in Production:** While random.Random(seed) in rng.py is perfect for your simulator and generating *.jsonl books, ensure that the actual production Stake Engine deployment strictly overrides this to use a Cryptographically Secure Pseudo-Random Number Generator.
### What I'd Add That You Didn't Ask About
 * **The Replay Viewer (Support Tooling):** A massive value-add for operators. Create a lightweight, separate entry point in Vite (e.g., ?mode=replay&payload=base64...) that accepts a raw Stake Engine spin JSON payload and plays it back. Customer support teams *require* this to resolve player disputes ("I swear I hit 5 scatters but the game crashed!").
 * **Centralized Pacing / Timeline Manager:** Quick Spin and Turbo Spin are mandatory player features today. Do not manage GSAP/Tween timings locally in individual symbol components. Create an AnimationManager singleton that scales the global timescale or drops non-critical tweens entirely based on a PacingState store.
 * **Audio Sprite Pipeline:** Mobile browsers strictly limit concurrent audio contexts. Loading individual .mp3 files via Howler will cause sound drops during intense features like cascading wins. Add a pipeline script to compile all game SFX into a single audio sprite file, using time-offsets to play specific sounds.
